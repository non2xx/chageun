# 디자인 시스템

프로젝트의 디자인 규칙(색·타이포·컴포넌트·레이아웃·UX 원칙)을 한 곳에서 관리하고, 새 UI를 만들거나 바꿀 때 기존 규칙과 맞는지 확인한다. 규칙이 없으면 현재 코드에서 추출해 초안을 만들거나, 새 규칙을 제안해 팀이 일관성을 유지하도록 돕는다.

**언제 쓰나:** 새 화면·컴포넌트·UI 요소를 추가하거나 기존 UI를 변경할 때, 디자인 규칙·스타일 가이드를 만들거나 확인할 때 발동한다. UI 작업이 아닌 순수 로직·API·DB 작업에는 사용하지 않는다.

---

## 0. 규칙 소스 탐색·해석 (항상 가장 먼저)

UI 작업을 시작하기 전, 프로젝트에 이미 디자인 규칙 파일이 있는지 먼저 확인한다.

**스캔할 흔한 이름·위치:**
- 마크다운 문서: `DESIGN.md`, `STYLEGUIDE.md`, `design-guidelines.md`, `docs/design*.md`
- 코드형 권위: `tailwind.config.*`, theme 파일, 토큰 파일(`tokens.js`, `variables.css`, `design-tokens.*`, `*.tokens.json`), styled-components/emotion `theme.ts`, MUI `createTheme`, Chakra theme, Vanilla Extract, Panda CSS 설정
- 컴포넌트 문서: `.storybook/` 디렉터리
- README 내 Figma 링크 (피그마가 공식 소스인 경우)

> **"못 찾음"을 성급히 결론짓지 말 것.** 위 목록에 안 잡혀도 `package.json`에 UI 프레임워크/디자인 라이브러리 의존성(MUI·Chakra·Tailwind·styled-components·Ant Design 등)이 있으면 이미 디자인 시스템이 있을 가능성이 크다. 초안을 새로 만들기 전에 "기존 디자인 시스템을 못 찾았는데 정말 없나요? 어디에 있나요?"라고 한 번 더 확인한다. (경쟁하는 두 번째 규칙 파일이 생기면 규칙이 갈라진다.)

> **오탐 가드 — 설계 문서 ≠ 디자인 규칙.** `2026-01-01-기능-design.md`처럼 이름 중간에 `design`이 든 파일은 보통 **브레인스토밍 spec(설계 문서)**이지 디자인 규칙이 아니다. 위 패턴 `docs/design*.md`는 `design`으로 **시작**하는 파일만 잡으므로 안전하지만, `*design*`처럼 넓게 매칭하지 말 것. 매칭된 파일을 채택하기 전 내용이 실제 디자인 규칙(토큰·컴포넌트·UX)인지 한 번 확인한다.

**상황별 처리 방식:**

- **파일이 있으면:** 그 파일을 진실의 원천(Single Source of Truth)으로 채택한다. 이름이 달라도 내용이 디자인 규칙이면 그걸 쓴다. 새 파일을 만들지 않는다.
- **부분만 덮는 파일이 있으면:** "빠진 층(예: 토큰은 있는데 UX 원칙이 없음)만 보완 문서로 만들까요?" 사용자에게 확인한 뒤 처리한다. 자동으로 추가 파일을 생성하지 않는다.
- **여러 파일이 있어 어느 게 기준인지 애매하면:** "어느 게 기준인가요?"라고 질문한다. 추측으로 하나를 고르지 않는다.
- **복합 케이스 (여러 소스가 각각 다른 층을 부분적으로 덮는 경우):** 순서를 고정한다.
  1. 먼저 "무엇을 기준 원천으로 삼을지" 사용자와 합의한다.
  2. 합의된 기준이 비워둔 층만 보완 문서로 추가한다.
  기준 합의 없이 보완부터 만들면 규칙이 갈라지므로 금지한다.
- **화면(UI)이 없는 제품이면:** CLI 도구·플러그인·라이브러리·API처럼 UI가 없는 제품은 디자인 규칙을 대조할 화면 자체가 없다. 규칙 파일도 UI 코드도 없으면 **"UI 없는 제품이라 디자인 대조 생략"을 한 줄로 보고하고 끝낸다**(억지로 규칙 문서를 만들지 않는다).
- **없으면(단, UI는 있는 제품):** 아래 `## 4. 규칙 파일이 없을 때`로 이동한다.

## 1. 디자인 규칙 4층 양식

규칙 파일을 새로 만들 때 사용하는 구조다. 기존 파일이 있으면 이 구조를 강제하지 않는다.

### ① 디자인 토큰

색·타이포·간격·라운딩 등 기본 값을 표로 정의한다.

| 이름 | 값 | 용도 |
|---|---|---|
| `color-primary` | `#2563EB` | 주요 버튼·링크·강조 색상 |
| `color-surface` | `#F8FAFC` | 카드·패널 배경 |
| `space-md` | `16px` | 섹션 내부 기본 여백 |
| `radius-card` | `8px` | 카드·모달 모서리 라운딩 |
| `font-body` | `14px / 1.6` | 본문 기본 폰트 크기·줄간격 |

토큰 이름은 역할 기반으로 짓는다(`blue-500` 대신 `color-primary`). 값이 바뀌어도 이름은 유지한다.

### ② 컴포넌트 패턴

자주 쓰는 UI 조각을 카드 형식으로 정의한다.

---
**버튼 (Button)**
- **언제 쓰나:** 사용자가 액션을 실행할 때 (저장, 제출, 삭제 등)
- **규칙:** 주요 액션 1개만 `primary` 스타일. 보조 액션은 `secondary` 또는 텍스트 버튼. 레이블은 동사로 시작 (예: "저장하기", "취소").
- **금지:** 한 화면에 `primary` 버튼 2개 이상 사용 금지. 레이블에 "클릭", "누르기" 같은 인터랙션 동사 사용 금지.

---

각 컴포넌트를 이 형식으로 추가한다. 예: 입력 필드, 카드, 모달, 내비게이션, 빈 상태(Empty State) 등.

**측정 가능한 스펙도 1급으로 적는다.** 컴포넌트 규칙엔 동작·금지뿐 아니라 **크기·높이·간격 같은 측정 가능한 값**을 명시한다 — "어떻게 써야 하나"만이 아니라 "어떻게 생겨야 하나". 예: *헤더 — `h1` 높이 `h-10` 고정(모든 페이지 동일).* **이렇게 문서에 적힌 측정 가능한 규칙이 빌드 점검·lint의 대상**이 된다. 거꾸로 **문서에 없으면 아무도 못 잡는다** — 헤더 높이가 페이지마다 달라도 "화면은 멀쩡히 뜨니" 통과돼 버리는 사고의 근본 원인이 이것이다.

### ③ 레이아웃·패턴

페이지 전체 구조와 간격 규칙을 서술한다.

- **페이지 구조:** 상단 글로벌 내비게이션 → 콘텐츠 영역(최대 너비 `1200px`, 좌우 여백 `24px`) → 하단 푸터(선택)
- **섹션 간격:** 주요 섹션 사이 `48px`, 카드 내부 `24px`, 인라인 요소 사이 `8~12px`
- **반응형:** 모바일 `< 768px`에서 단일 컬럼, 태블릿 `768px~1024px`에서 2컬럼, 데스크톱 `> 1024px`에서 최대 3컬럼
- **내비 위치:** 상단 고정(`sticky`). 모바일에서는 하단 탭 바로 대체 가능.

예시: 목록 화면은 `헤더(제목 + 주요 액션 버튼)` → `필터·검색 바` → `목록 카드` 순서로 배치한다.

### ④ UX 원칙·톤앤매너

글쓰기 방식, 접근성, 오류·빈 상태 처리 원칙을 정의한다.

- **보이스·라이팅:** 짧고 명확한 문장. 전문 용어 대신 일상 언어 사용. 오류 메시지는 "문제 + 해결 방법" 형식으로 (예: "이메일 형식이 맞지 않아요. 다시 확인해 주세요.").
- **접근성:** 모든 이미지에 대체 텍스트 제공. 색만으로 정보를 전달하지 않음(아이콘·텍스트 병행). 포커스 링 항상 표시.
- **오류 상태:** 인라인 오류는 해당 필드 바로 아래에 빨간 텍스트로 표시. 전체 오류는 폼 상단 배너.
- **빈 상태(Empty State):** "아직 항목이 없어요" + 안내 일러스트 + 주요 액션 버튼 1개. 빈 화면을 그냥 두지 않는다.

## 2. 동작

### 읽기
UI 작업을 시작할 때 `## 0`의 절차대로 규칙 소스를 탐색하고 로드한다. 규칙을 찾으면 해당 내용을 참고해 이후 작업을 진행한다.

### 검증
UI를 만들거나 변경한 뒤 **빌드·실제 구동 검증·끝 점검** 단계에서 화면과 규칙을 대조한다.
- 규칙에 맞으면 통과.
- 규칙과 다르면 어떤 규칙을 어떻게 어겼는지 명시하고, 수정 또는 예외 처리를 제안한다(아래 `## 3` 3택).

**화면(IA 노드)마다 "디자인 준수" 점검을 기능 점검과 나란히 둔다.** 기능 검증 체크리스트가 *"작동하나"*(진입/동작/영향)를 본다면, 디자인 점검은 *"문서 규칙대로 생겼나"*(헤더 높이·간격·토큰·컴포넌트 규칙)를 본다 — **직각(orthogonal) 축**이다. 기능이 완벽해도 외형이 규칙과 다를 수 있으니 둘은 별개로 본다.

**디자인 점검은 빌드·검증 시점에만 — 매일 정기 점검(`monitoring`)엔 넣지 않는다.** 외형 일관성은 runtime이 아니라 **만들 때** 잡는다(매일 화면을 다시 비교하는 건 무겁고 거짓경보가 잦다). 디자인 규칙은 **design-system 문서가 단일 원본** — 작업 카드에서 휘발시키지 않고 문서에 저장·갱신한다(기능의 검증 체크리스트 등뼈와 같은 방식).

### 갱신·제안
새 패턴이 발견되면 아래 `## 3`의 절차를 따른다.

---

## 3. 새 UI → 새 규칙 제안 루프

규칙에 없는 새 UI 패턴이 필요할 때 아래 3택을 사용자에게 제안한다.

> 이 패턴은 현재 디자인 규칙에 없어요. 어떻게 할까요?
> 1. 기존 규칙에 맞추기 — 가장 가까운 기존 패턴을 적용합니다.
> 2. 새 규칙으로 등록 — 이번 패턴을 디자인 규칙 문서에 추가합니다.
> 3. 일회성 예외 — 이번 한 번만 예외로 처리하고 문서는 바꾸지 않습니다.

**②번 선택 시:** 사용자가 승인한 뒤에만 규칙 문서에 추가한다. 변경된 부분만 간결하게 갱신하며, 파일 전체를 다시 쓰지 않는다.

---

## 4. 규칙 파일이 없을 때

규칙이 없어도 UI 작업을 막지 않는다. 다음 두 가지를 제안한다.

> 디자인 규칙 문서가 없어요. 어떻게 할까요?
> 1. 코드에서 현재 색·컴포넌트를 추출해 초안 만들기 — 기존 코드를 스캔해 `docs/design-system.md` 초안을 만듭니다.
> 2. 이번엔 그냥 진행 — 규칙 없이 작업하고 나중에 문서화합니다.

**①번 선택 시:** 현재 코드(`tailwind.config`, CSS 변수, 컴포넌트 파일 등)를 스캔해 `docs/design-system.md` 초안을 만든다. 기본 파일 경로는 새로 만들 때만 `docs/design-system.md`를 사용한다. **그 경로에 파일이 이미 있으면 덮어쓰지 않고 사용자에게 확인**한다(다른 용도의 파일일 수 있음).

**②번 선택 시:** "규칙 문서 없어 대조 생략"을 한 줄로 보고하고 작업을 계속한다.

### 규칙도 없고 디자인이 들쭉날쭉/하드코딩일 때 (브라운필드)

코드가 **규칙 없이 제각각이고 색·크기가 하드코딩**이면, 깔끔한 추출이 불가능하다(추출해봤자 규칙이 아니라 난장판이 나온다). 일괄로 갈아엎지 말고 — **인벤토리 → 기준 → 점진 수렴** 순서로 간다(product-map 브라운필드와 같은 패턴).

1. **인벤토리(현황 가시화):** 버튼·색·간격·헤더 높이가 **각각 몇 종 쓰였는지 목록으로** 보여준다(예: "버튼 8종, 직접 색상 23개, 헤더 높이 5종류"). 숨은 혼란을 눈에 보이게 만드는 것만으로 가치가 있다.
2. **기준 정하기:** 합리적 기준을 **제안**하거나 사용자가 **고른다**(예: "가장 많이 쓰인 파랑을 `color-primary`로?"). 그 결정을 `docs/design-system.md` 초안에 적어 **단일 원본**으로 삼는다.
3. **점진 수렴:** **새 작업부터 기준 적용 / 기존 화면은 건드릴 때(touch) 정렬 / 아직 안 맞춘 곳은 `(미정렬)` 표시 / lint는 처음엔 warn-first**(난장판이 작업을 막지 않게). **한 번에 일괄 재생성 금지** — 위험하고, "있는 걸 버리지 않고 점진적으로 다듬는다"는 원칙에 어긋난다.

chageun는 난장판을 **자동으로 통일해 주지 않는다**(어느 스타일이 정답인지는 사용자의 디자인 결정). **보이게·결정 가능하게·점진적으로** 만들어 줄 뿐이며, 한 번에 다 고치라고 강요하지 않는다.

## 5. 정적 검사(lint) 연결

디자인 문서의 **측정 가능한 규칙**(헤더 높이·간격·금지 클래스 등)을 그 프로젝트의 **정적 검사(lint) 규칙으로 생성·확장**하도록 안내한다. **문서에 규칙 추가 = lint가 자동으로 enforce** — 문서와 lint가 끊겨 있으면(예: lint가 색·글자 크기만 보고 헤더 높이 규칙이 없으면) 규칙을 적어도 아무도 못 잡는다. 이 끊김을 잇는 게 핵심이다.

- **A안(인프라 미동봉):** chageun는 lint 엔진을 품지 않는다 — 그 프로젝트의 lint(ESLint·Stylelint·커스텀 규칙)에 규칙을 **생성·확장**하고, **CI(push/PR)에서 돌리도록** 안내한다(`security-scan`과 같은 틀).
- **warn-first → error 승격:** 브라운필드 난장판이면 처음엔 **경고만**(작업을 막지 않음), 수렴되면 **error로 승격**한다.
- **거짓경보는 심각도 높은 것부터**(다 잡으려다 지치지 않게 — 노이즈 병목).
- **토큰을 기계가 읽게 두면 자동 검사가 쉬워진다:** 색·타이포·간격 토큰을 사람이 읽는 표뿐 아니라 **기계 가독 형식(YAML 등)** 으로도 두면 lint·AI가 값을 직접 읽어 enforce한다(이름만 다른 색 23종, 미정의 토큰 참조 등을 잡음). **명도비(WCAG 대비)** 도 자동 점검 대상에 넣는다(접근성). 이 "토큰 + lint"를 한 벌로 주는 기성 표준으로 **`design.md`**(Google Labs)가 있으니, 원하면 그 형식/도구를 **한 옵션으로 안내**한다 — 단 alpha라 하드 의존하지 말고 선택지로만.
