# 정기 점검 (monitoring)

손님이 말하기 전에 깨짐을 **자동으로 인지**하기 위해, 검증 체크리스트(끝 점검에서 feature-spec에 쌓인 `[정기 점검 대상]`)를 **매일 자동으로 실행**한다. 깨지면 알림 + chageun가 쉬운 말로 원인·수정을 제안한다.

chageun는 **점검을 직접 돌리는 인프라를 품지 않는다.** 발동될 때 그 프로젝트에 맞는 점검 설정을 **생성·안내**할 뿐이다(세션이 닫혀도 점검이 돌도록 GitHub Actions 같은 외부 스케줄러에 얹는다).

---

## 언제 쓰나 / 안 쓰나

**발동:** 끝 점검에서 `[정기 점검 대상]`이 생겼고, **운영(배포)되는 DB·UI 앱**일 때. 사용자가 "자동 점검 붙여줘"라고 할 때.

**비발동:** 정적 사이트·읽기 전용·일회성 스크립트, 점검이 이미 붙어 있을 때. 억지로 권하지 않는다.

---

## 0. 전제조건 점검 (분기 — 먼저 한다)

상황을 먼저 확인하고, 안 맞으면 빠지는 길로 간다.

| 상황 | 길 |
|---|---|
| **GitHub + Docker로 앱이 띄워짐** | **본 코스** (아래 1~3) |
| **GitHub 아님**(로컬·GitLab 등) | **대안 A** — 로컬에서 도는 점검 스크립트를 만들고, 수동 실행 또는 로컬 스케줄러(cron/작업 스케줄러)로 돌리도록 안내 |
| **DB 없음·정적·읽기 전용** | **대안 B** — 무거운 Docker 안 세움. 가벼운 "살아있나(켜져 있나)" 핑만 |
| **백엔드가 클라우드 SaaS라 Docker로 못 띄움** | 운영 아닌 **별도 테스트 프로젝트/스테이징만** 연결(운영 금지 — operating-rules "클라우드 백엔드는 로컬 복제본/별도 테스트 프로젝트만"과 정합). 그게 없으면 **검증 보류** |
| **격리 테스트 환경이 없음**(운영=테스트 구분 안 됨) | **검증 보류** — "동작 검증 안 됨 — 미검증 출시 위험"으로 보고만. **운영에서 점검하지 않는다(hard block).** |

모노레포면 점검 대상 패키지/제품의 `docs/` 위치를 먼저 확인한다.

---

## 1. 점검 항목 출처

`docs/feature-spec.md`의 `[정기 점검 대상]` 검증 체크리스트(IA 1-hop 연결 포함)를 **읽기만 한다 — 새로 만들지 않는다**(단일 출처). 우선순위 높은 기능만, **하루 1번**. 체크리스트의 **진입 / 동작 / 영향 / 엣지**가 그대로 점검 시나리오가 된다.

**빈도는 하루 1번이 기본 — 더 자주 돌리지 않는다.** 시간당처럼 자주 관측하면 신호보다 **잡음(거짓 경보)** 이 급격히 늘어 정작 진짜 문제를 못 본다(노이즈 병목). 자주 봐야 할 진짜 이유가 없으면 하루 1번을 지킨다.

**출처 없음 분기:** `docs/feature-spec.md`가 없거나 `[정기 점검 대상]`이 0개면 점검을 만들지 말고 **"먼저 `product-map`으로 검증 체크리스트부터 만들까요?"** 라고 안내한다. 빈 점검을 생성하지 않는다.

---

## 2. 생성 (그 프로젝트에)

**이 격리 환경(harness) 구축은 1회성·전부 아니면 전무다.** 반쪽짜리 harness는 아무것도 검증 못 한다 — "지을까/말까"를 정하는 **한 번의 결정**이고, **지으면 전 기능 자동 점검이 목표**(화면 하나만 하고 멈추는 게 아니다). 일단 서면 화면 추가 비용은 작다. **계속 드는 비용은 단 하나: 스키마/마이그레이션이 바뀌면 씨앗·stub도 같이 갱신**(harness는 한 번 짓고 끝이 아니라 유지하는 살림). 이 점을 솔직히 알리되, **"어려우니 하지 말자"가 아니라 "한 번 지으면 전부 자동으로 돈다"** 로 안내한다(겁줘서 발 빼게 만들지 않는다).

본 코스에서 다음을 그 프로젝트에 맞춰 생성한다.

- **`.github/workflows/daily-check.yml`** — 트리거는 **`schedule`(cron 하루 1번) + `workflow_dispatch`(수동 버튼)만.** `pull_request`·`pull_request_target`은 쓰지 않는다(fork PR에서 시크릿이 새는 함정 차단).
- **`docker-compose.test.yml`** — 앱 + **일회용 테스트 DB**(점검이 끝나면 버려지는 깨끗한 DB). 로컬 복제본(예: 로컬 Supabase)으로 운영과 완전 분리.
- **씨앗 데이터(seed) 스크립트** — 빈 테스트 DB로는 아무 흐름도 안 돈다. 앱이 기대하는 **최소 데이터**(회사·역할/권한·기준 거래처·기준 제품·필수 설정 등)를 심는 `seed.sql`(또는 동등물)을 만든다. **반드시 가짜(합성) 데이터로 — 운영 PII(실제 고객 이름·주소·이메일·전화)를 테스트 DB에 복사하지 않는다.** **마이그레이션이 바뀌면 씨앗도 함께 갱신**(1회성 아님 — 위 유지비용의 핵심).
- **점검 스크립트** — 체크리스트를 실제로 실행(진입→동작→영향, DB 입력까지). 도구는 프로젝트 성격에 맞춰 **스킬이 선택**한다 — 웹 화면이면 **Playwright**(CLI보다 **MCP 방식이 더 안정적** — 실패율이 절반 수준이라 MCP 우선), API면 **HTTP 호출**. 거짓 실패(flaky)를 줄이려 **2회까지 재시도**하고 그래도 실패면 실패로 본다.

### 외부로 나가는 연동 처리 (stub / 리다이렉트)

DB가 아니라 **바깥 세상으로 나가는 동작**(이메일·세금계산서·문자·결제·배송 등)은 로컬 테스트 DB로도 못 막는다 — 점검에서 진짜로 부르면 **진짜 발송·진짜 발행**이 일어난다. **기본은 가짜(stub)로 막는다.**

- **기본(stub):** 외부 호출을 가로채 가짜 응답. 커버리지는 "**앱이 DB에 기록까지**"(예: 발송 기록·계산서 데이터 생성). "진짜로 나갔나"는 범위 밖 — 이 한계를 **숨기지 말고 명시**한다.
- **렌더링을 눈으로 봐야 하는 출력(이메일 템플릿 등):** ① **안 보내고 확인** — 템플릿을 HTML로 렌더해 스냅샷/스크린샷(Playwright). 변수 채움·레이아웃 확인엔 충분, 가장 안전. ② **실제 발송 경로까지** — **이메일 캐처/샌드박스(Mailpit·Mailtrap 등)** 로 보낸다(진짜 사람에게 안 가고 렌더 결과를 봄). 또는 **수신자를 테스트 주소 1개로 하드 고정.**
- **수신자 고정은 반드시 설정(env)에서 강제 — 씨앗/DB 데이터에 의존 금지.** 씨앗은 가짜 주소만 쓰지만, 그럼에도 **설정이 무조건 테스트 주소로 덮어쓰게**(데이터에 기대지 않는 방어 다층).
- **fail-closed가 기본값(필수):** stub·수신자 오버라이드 가드가 **켜진 게 확인 안 되면 발송을 막고 점검을 실패 처리**한다. 발송 라이브러리의 통상 기본값은 "설정 없으면 실제 발송"이라, 그걸 **뒤집어** "가드 미확인 시 발송 금지"로 둔다. "설정됐겠지"를 신뢰하지 않고 점검 시작 시 가드 상태를 확인한다.
- **되돌릴 수 없는 발송(세금계산서·결제·문자 등)은 테스트 주소로도 실제 호출 금지** — 순수 stub 또는 공식 샌드박스만. (수신자만 바꿔도 행위 자체가 비가역이라 안전해지지 않는다.)
- **매일 자동 점검엔 캐처/샌드박스를 우선**(매일 실제 발송 = 한도·받은편지함 노이즈·유출 위험). 실제 발송(②)은 **되돌릴 수 있는 출력에 한해**(예: 내 받은편지함으로 가는 이메일) 가끔 수동 확인용으로만 — **비가역 발송엔 절대 적용하지 않는다.**

### 시크릿 가드 (필수)

- GitHub Secrets에는 **테스트/스테이징 키만 — 운영 키는 절대 금지.**
- DB 접속은 반드시 **일회용 테스트 DB.** 운영 DB 연결 문자열을 넣지 않는다. *(이 지점이 "운영 쓰기 hard block"이 우회되는 함정이다.)*
- 시크릿 값은 chageun가 보거나 저장·커밋하지 않는다 — "GitHub Secrets에 직접 넣으세요"만 안내한다.

---

## 3. 깨지면

- **GitHub 기본 실패 메일**이 저장소 주인에게 자동으로 간다(무설정).
- 다음에 chageun를 부르면 실패 로그를 읽어 `systematic-debugging`으로 **쉬운 말 원인·수정**을 제안한다(추측 금지 — 재현→원인→수정).

### 비개발자가 알아야 할 한계 (쉬운 말로 안내한다)

- **가끔 거짓 경보(flaky)** 가 날 수 있다 → 그래서 점검은 **2회 재시도** 후 판정한다.
- 실패 메일은 **저장소 주인 1명에게만** 간다 → 팀이 함께 받으려면 별도 채널 추가가 필요(백로그).
- GitHub은 **커밋이 60일 없으면 자동 점검(cron)을 꺼버린다** → 가끔 커밋하거나 Actions 탭에서 살아 있는지 확인한다.

---

## 4. (선택) 운영 URL 핑

운영 주소가 **"살아 있나(켜져 있나)"** 만 확인하는 가벼운 핑 1개. **쓰기 금지** — 데이터를 건드리지 않는다. 핑은 **GET/HEAD만, 상태코드(200 등)만** 본다 — 데이터를 남기는 엔드포인트(POST 등)는 호출하지 않는다.

---

## 5. 많아지면

저장소마다 GitHub Actions가 기본이다. 프로젝트가 너무 많아지면 무료 가벼운 감시툴(UptimeRobot류)로 "살아있나"를 한곳에서 보는 정도만 안내한다 — **작게.** 풀 모니터링 도구를 세우지 않는다.

**무료 한도 주의(쉬운 말로 안내):** 비공개 저장소는 GitHub Actions 무료 시간(월 약 2,000분)이 있다 — 점검이 많아 한도를 넘기면 **그 달엔 점검이 조용히 안 돈다.** 이럴 땐 무거운 Docker 점검을 줄이고 가벼운 핑으로 축소하도록 안내한다.

---

## 안전 (필수 — operating-rules와 정합)

- **운영 DB 쓰기 검증은 hard block — 사용자 동의로도 우회 불가.** 테스트 DB·테스트 키만 쓴다. "운영에서 해볼까요?"라고 묻지 않는다.
- 격리 환경(Docker 테스트 DB)은 **DB·쓰기 백엔드가 있는 앱에만** 세운다. 정적·읽기 전용은 핑으로 충분.
- 시크릿은 저장·커밋하지 않고 GitHub Secrets만 가리킨다.
